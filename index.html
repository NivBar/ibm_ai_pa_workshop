Workshop architecture

Planning Analytics stays at the center

AI makes governed models and business logic easier to understand and use.

Planning Analytics

Owns the governed cubes, dimensions, measures, hierarchies, rules, feeders, processes, data, plans, and calculations.

MCP

Model Context Protocol exposes approved Planning Analytics tools through a consistent interface.

AI agent

Reasons about the request, selects MCP tools, interprets results, and assembles an answer within its instructions.

IBM Bob

The primary hands-on workspace used to inspect and investigate Planning Analytics throughout the workshop.

watsonx Orchestrate

The optional final session demonstrates the same end-to-end investigation as one unified agent interaction.

Bob or watsonx Orchestrate→AI agent→ibm-pa-tools MCP→Planning Analytics / NewRetail→Grounded output
One-prompt workflow

For each activity, copy its single prompt into Bob. Bob queries Planning Analytics and directly extends NewRetail_Model_Health_Dashboard.html. There is no paste-back, note-taking, completion control, or separate export step.

Evidence rule

Never accept a fluent answer as proof. Validate cube data and coordinates, dimensions, hierarchies, rules, feeders, processes, dependencies, logs, metrics, outliers, impact results, and returned system state.

Module 01

Connect and orient

Confirm the live tool boundary before asking business questions.

2 activities

Business context: You are entering an unfamiliar Planning Analytics environment and need to prove which system and tools will ground the investigation.

Activity 1

Verify the MCP connection

Objective Confirm that Bob can call the configured Planning Analytics MCP server and that NewRetail is available.

Available input

The preconfigured or copied MCP connection below.

Do this

Ask for available servers; verify NewRetail; record the tool result rather than assuming connectivity.

Prompt

What TM1 servers are available through the ibm-pa-tools MCP connection? Confirm whether NewRetail is available and state which tool supplied the answer.

Expected output

A verified server result or a precise connection error.

Activity 2

Inventory the tool surface

Objective Understand what the connected server can inspect, analyze, and operate safely.

Available input

The live MCP tool catalogue.

Do this

Group tools into metadata, data analysis, process/rule, logs/metrics, impact/outlier, and write-capable categories. Do not invoke write-capable tools.

Prompt

List the Planning Analytics MCP tools available in this session. Group them by discovery, querying, process inspection, operational diagnostics, AI analysis, and model-changing operations. Mark the last group as out of scope.

Expected output

A capability map and explicit read-only boundary.

Contributes to the final dashboard → Connection evidence and system scope

Module 02

Understand the Planning Analytics environment

Map the governed components before analyzing values.

4 activities

Business context: A new analyst must understand what exists, how it is organized, and which model is appropriate.

Activity 1

Discover cubes

Objective Identify the commercial and planning cubes relevant to the investigation.

Available input

NewRetail server metadata.

Do this

List cubes with metadata; distinguish Revenue, Income Statement, Metrics, and other potentially related models.

Prompt

On NewRetail, list cubes with metadata. Identify which cubes may answer questions about revenue, profitability, plans, forecasts, and operational metrics. Explain the evidence for each choice.

Expected output

A concise cube inventory with evidence-based relevance.

Activity 2

Inspect the Revenue model

Objective Understand its dimensional grain and available measures.

Available input

Revenue cube metadata and sample members.

Do this

Inspect dimensions, hierarchy roles, sample members, versions, periods, products, organizations, and channels.

Prompt

Inspect the Revenue cube dimensions with metadata and sample members. Explain the analytical role of each dimension, identify measures and versions, and note any ambiguous labels that should be validated before querying.

Expected output

A model dictionary suitable for a new analyst.

Activity 3

Compare related models

Objective Find shared dimensions and possible cross-cube joins.

Available input

Metadata for Revenue and at least one related cube.

Do this

Compare dimensionality, shared members, measures, and grain; do not infer a relationship solely from similar names.

Prompt

Compare the Revenue cube with the most relevant related cube. Show shared dimensions, different measures, level of detail, and which cross-cube comparisons are valid or unsafe.

Expected output

A cross-cube compatibility assessment.

Activity 4

Inspect AI metadata

Objective Understand which cubes are analyzed for natural-language use.

Available input

AI-analysis metadata returned by MCP.

Do this

List AI-analyzed cubes and explain what the metadata does and does not guarantee.

Prompt

Which NewRetail cubes have AI-analysis metadata? Summarize their descriptions and explain why AI-ready metadata does not replace validation of dimensions, members, and business logic.

Expected output

An AI-readiness inventory with limitations.

Contributes to the final dashboard → Environment inventory and model overview

Activity 1

Inventory TurboIntegrator processes

Objective Identify processes that load, transform, copy, or maintain relevant data.

Available input

Live NewRetail process inventory.

Do this

List processes and categorize them by likely purpose, labeling name-based interpretations as hypotheses.

Prompt

List the TurboIntegrator processes on NewRetail. Group them into data load, copy/forecast, dimension maintenance, and administrative categories. Separate verified process details from name-based hypotheses.

Expected output

A categorized process inventory.

Activity 2

Inspect a relevant process

Objective Explain inputs, parameters, logic, targets, and error handling.

Available input

Process details for a relevant Revenue or maintenance process.

Do this

Select a relevant process from the inventory and inspect its full definition; do not execute it.

Prompt

Inspect the full details of the most relevant process for the Revenue investigation. Explain its parameters, data source, Prolog, Metadata, Data, and Epilog behavior, target objects, dependencies, and error handling in plain language. Do not execute it.

Expected output

A technical and business-readable process explanation.

Activity 3

Trace a calculated metric

Objective Determine whether a selected metric is stored, aggregated, or rule-calculated.

Available input

Cube metadata, rule information available through connected tools, and process details.

Do this

Trace the selected metric through rules, feeders, source values, and load processes; identify evidence gaps.

Prompt

Trace how one important Revenue metric is calculated or populated. Distinguish stored input, consolidation, rule calculation, feeder behavior, and process-loaded values. Cite returned metadata or code and label anything the tools cannot verify.

Expected output

A metric lineage with verified and unknown steps.

Activity 4

Build the dependency map

Objective Connect cubes, dimensions, processes, rules, files, and outputs.

Available input

All verified outputs from this module.

Do this

Create a directional dependency map; every edge must have evidence or a hypothesis label.

Prompt

Build a dependency map for the Revenue model showing source inputs, TurboIntegrator processes, shared dimensions, cubes, rules/feeders, saved views, and downstream reporting. For every edge, state the evidence or mark it as unverified.

Expected output

An auditable model dependency map.

Contributes to the final dashboard → Dependency map and calculation lineage

Module 04

Investigate business performance

Move from headline variance to multidimensional evidence.

5 activities

Business context: The latest completed planning period contains a material Actual-versus-plan gap that needs explanation.

Activity 1

Establish the current baseline

Objective Find the latest period with comparable Actual and plan data.

Available input

Live Revenue data; no fixed period assumption.

Do this

Determine the latest complete comparable period, then query Actual, Budget/Plan, variance amount, and variance percent.

Prompt

Using the Revenue cube, determine the latest complete period with both Actual and Budget or Plan data. Return total revenue, variance amount, and variance percent. State the exact members and filters used.

Expected output

A reproducible baseline tied to explicit cube coordinates.

Activity 2

Decompose the variance

Objective Find where the headline gap originates.

Available input

The verified baseline.

Do this

Break the variance across organization/region, product, channel, and time; rank contributions rather than only totals.

Prompt

For the verified period, decompose the Actual-to-plan revenue variance by organization, product category, channel, and month. Rank positive and negative contributors and show how each contributes to the total variance.

Expected output

A ranked driver table with reconciliation to the headline.

Activity 3

Analyze trends and outliers

Objective Distinguish normal movement from unusual observations.

Available input

Monthly data and MCP outlier detection.

Do this

Query a stable time series, run outlier detection, and validate flagged points against business dimensions.

Prompt

Show the longest reliable monthly Actual revenue series available for the selected slice, then use the MCP outlier capability to identify unusual observations. Explain the algorithm result, validate each candidate against surrounding periods, and avoid treating statistical outliers as proven business issues.

Expected output

A trend assessment and validated anomaly candidates.

Activity 4

Compare related cubes

Objective Test whether the pattern appears in revenue, profitability, or another governed model.

Available input

The compatibility assessment from Module 2.

Do this

Run a valid cross-cube comparison using shared dimensions and consistent periods.

Prompt

Compare the selected Revenue pattern with the most relevant measure in a compatible related cube. Align period, organization, product, and version members explicitly. Explain agreements, contradictions, and grain differences.

Expected output

A governed cross-cube comparison.

Activity 5

Create a directional outlook

Objective Use observed trends without confusing an AI estimate with the governed plan.

Available input

Verified time series and current plan values.

Do this

Generate a simple directional outlook; state method, assumptions, and limitations; compare with the governed plan.

Prompt

Using the verified monthly series, create a clearly labeled directional outlook for the next planning period using a simple transparent method. Compare it with the governed Planning Analytics plan and explain assumptions, uncertainty, and why this estimate does not replace the plan.

Expected output

Forecast context with clear governance boundaries.

Contributes to the final dashboard → Financial findings, trend, forecast context, and comparison evidence

Module 05

Investigate and explain an anomaly

Separate observation, hypothesis, evidence, and conclusion.

4 activities

Business context: The analysis isolated an unexpected product-organization-period result that may reflect business change, master data, calculation logic, or incomplete loading.

Activity 1

Reproduce the anomaly

Objective Confirm it persists at explicit coordinates.

Available input

The selected anomaly and its exact cube slice.

Do this

Requery exact coordinates; compare totals and leaf levels; test whether consolidation hides the effect.

Prompt

Reproduce the selected anomaly using explicit cube, dimension, hierarchy, member, version, and period coordinates. Compare consolidated and leaf-level results and report whether the anomaly is stable.

Expected output

A reproducible observation, not a narrative assumption.

Activity 2

Test competing explanations

Objective Evaluate several plausible causes systematically.

Available input

Cube values, hierarchy metadata, process and rule evidence.

Do this

Test business movement, hierarchy/master-data change, calculation behavior, missing input, and cross-cube inconsistency.

Prompt

Build and test a root-cause tree for the anomaly. Evaluate business movement, hierarchy or attribute change, rule/calculation behavior, missing or malformed input, and cross-cube inconsistency. For each branch, state supporting evidence, contradicting evidence, and the next test.

Expected output

A ranked hypothesis table.

Activity 3

Use impact analysis

Objective Identify dimensions or cells most associated with the selected result.

Available input

An analytical state returned by a relevant data-explorer query.

Do this

Run MCP impact analysis on a bounded slice; interpret associations carefully.

Prompt

Run Planning Analytics impact analysis on the bounded slice surrounding the anomaly. Summarize the strongest drivers, explain what the scores mean, and state why association alone does not prove causation.

Expected output

A driver assessment with methodological caution.

Activity 4

Write the conclusion

Objective Explain the result at technical and executive levels.

Available input

All verified anomaly evidence.

Do this

Produce a short technical finding and an executive explanation; preserve unknowns.

Prompt

Summarize the anomaly in two layers: a technical finding with coordinates and dependency evidence, then a concise executive explanation. Clearly label what is proven, likely, and still unknown.

Expected output

A defensible root-cause statement.

Contributes to the final dashboard → Root-cause tree and confidence assessment

Module 06

Operate and troubleshoot

Use live system evidence before proposing recovery.

5 activities

Business context: A process appears not to have produced the expected result. Diagnose safely without executing or changing anything.

Activity 1

Check process status and threads

Objective Determine whether a relevant process is running, queued, completed, or absent.

Available input

Live process status and thread tools.

Do this

Inspect process status and server threads; do not cancel or execute anything.

Prompt

Check the current status of the relevant NewRetail process and inspect active process threads. Explain whether the evidence shows a running, queued, completed, or inactive process. Keep the investigation read-only.

Expected output

A current operational state assessment.

Activity 2

Inspect execution logs

Objective Find actual errors or confirm that no error evidence exists.

Available input

Live process execution error logs.

Do this

Retrieve logs for the selected process; distinguish empty results, informational history, warnings, and errors.

Prompt

Retrieve the available execution error logs for the selected NewRetail process. Build a short timeline of relevant entries, classify severity, identify parameters or input issues, and explicitly state when the logs do not prove a failure.

Expected output

An evidence-based log interpretation.

Activity 3

Diagnose input quality

Objective Test whether missing or malformed input could explain the symptom.

Available input

Process definition, expected source structure, and live logs.

Do this

Compare expected source/header/parameters with observed evidence; do not fabricate file contents.

Prompt

From the inspected process definition and logs, determine the expected input file, fields, parameters, and preconditions. Identify evidence of a missing file, malformed header, invalid member, or rejected row. If source contents are unavailable, list the exact validation still required.

Expected output

An input validation checklist grounded in process logic.

Activity 4

Assess runtime and memory

Objective Look for performance bottlenecks using live metrics and code evidence.

Available input

TM1 server metrics, process details, threads, and rule/feeder evidence.

Do this

Check runtime, memory, locking, cell-by-cell writes, broad feeders, and oversized query/view patterns; separate observed signals from administrator-only checks.

Prompt

Review available NewRetail server metrics, process code, thread evidence, and rule or feeder information for runtime, memory growth, lock contention, cell-by-cell writes, broad feeders, or oversized query patterns. Separate observed warning signs from checks that require administrator telemetry.

Expected output

A performance risk register with evidence.

Activity 5

Recommend safe recovery

Objective Define validation, recovery, prevention, and escalation criteria without executing changes.

Available input

The complete operational evidence set.

Do this

Sequence read-only validation, controlled recovery, reconciliation, and prevention; specify where approval is required.

Prompt

Produce a safe recovery plan for the diagnosed issue: prechecks, controlled rerun or repair steps, post-run reconciliation, rollback criteria, preventive controls, and actions requiring explicit administrator approval. Do not execute anything.

Expected output

An actionable recovery plan with control gates.

Contributes to the final dashboard → Operational diagnosis and recovery recommendations

Module 07

Assess a proposed model change

Use dependency knowledge before modifying the model.

4 activities

Business context: The business proposes adding a new product category and wants the full downstream impact understood first.

Activity 1

Locate the insertion points

Objective Explain exactly where a new element would enter the model.

Available input

Verified product hierarchy, attributes, process parameters, and mappings.

Do this

Inspect Add_Product and related metadata; identify parent, code, alias, attributes, source mapping, and permissions.

Prompt

Inspect the NewRetail product hierarchy and the Add_Product process. Explain the information required to add a new product category, including parent, unique code, caption or alias, attributes, source mappings, and permissions. Do not execute the process.

Expected output

A read-only element-addition procedure.

Activity 2

Map downstream impact

Objective Identify every verified consumer of the changed dimension.

Available input

Dependency map plus process, rule, cube, and view metadata.

Do this

Assess cubes, rules/feeders, TI, views, subsets, security, reports, and cross-cube compatibility.

Prompt

Assess the downstream impact of adding a product category. Cover affected cubes, consolidations, attributes, rules/feeders, TurboIntegrator processes, source mappings, subsets, saved views, security, AI metadata, and reporting. Cite verified dependencies and flag unknowns.

Expected output

A dependency-based change impact matrix.

Activity 3

Test the proposed design

Objective Define positive, negative, reconciliation, performance, and rollback tests.

Available input

The impact matrix and existing model behavior.

Do this

Create a test matrix without performing writes.

Prompt

Create a test plan for the proposed product-category change. Include hierarchy validation, data loading, rule calculations, feeder behavior, consolidation reconciliation, cross-cube consistency, security, report behavior, performance, negative cases, and rollback evidence.

Expected output

A controlled test and rollback plan.

Activity 4

Prioritize change risks

Objective Turn technical dependencies into a release decision.

Available input

Impact and test findings.

Do this

Rank risks by likelihood, impact, detectability, and mitigation; identify go/no-go gates.

Prompt

Rank the proposed change risks by likelihood, business impact, detectability, and mitigation. Define explicit go/no-go gates and the evidence required for approval.

Expected output

A decision-ready change recommendation.

Contributes to the final dashboard → Change impact matrix, test plan, and maintenance guidance
